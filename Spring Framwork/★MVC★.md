# MVC(Model, View, Controller) Pattern

## M V C로 나누는 이유

```
1. 하나의 클래스에 로직을 다 작성해 버리면 협업이 안됨.
2. 프로그래밍 로직을 서로 영향 없이 쉽게 고칠 수 있는 소프트웨어 디자인 패턴
```

## MVC 각각의 의미

```
M : Model (@Service와 DB 두가지로 나뉨!)
    1. 비즈니스 로직(@Service(로직 작성하는 클래스에 작성))
    2. DB 연동 로직 (MyBatis)

V : View
    최종 사용자에게 결과를 화면(UI)으로 보여줌

C : Controller
    1. url로 요청을 받아서 화면과 Model을 연결 시켜주는 역할(Controller에서는 로직을 구현하지 않는다, 로직을 작성한 Model을 return할 뿐)
    2. Model의 로직을 받아 View로 전송
```

## MVC 개념

```
1. Model
    1-1) 어플리케이션의 정보, 데이터를 나타냄
    1-2) 데이터와 비즈니스 로직을 관리, 애플리케이션이 포함해야할 데이터가 무엇인지를 정의
    1-3) 모델이 갖춰야할 규칙
        1-3-1) 사용자가 편집하길 원하는 모든 데이터를 가지고 있어야한다.
        1-3-2) 뷰나 컨트롤러에 대해서 어떠한 정보도 알면 안됨.
        1-3-3) 변경이 일어나면, 변경 통지에 대한 처리 방법을 구현해야한다.

2. View(사용자에게 보여지는 부분 == 레이아웃과 화면을 처리 == 유저 인터페이스를 의미)
    2-1) 모델에게 전달받은 데이터를 별도로 저장 X
    2-2) 사용자가 화면에 표시된 내용을 변경하게 되면 모델에게 전달하여 모델을 변경
    2-3) 애플리케이션의 데이터를 보여주는 방식을 정의
    2-4) 뷰가 갖춰야할 규칙
        2-4-1) 모델이 가지고 있는 정보를 따로 저장해서는 안됨.
        2-4-2) 뷰는 모델이나 컨트롤러와 같이 다른 구성요소들을 몰라야함.
        2-4-3) 변경이 일어나면 변경통지에 대한 처리방법을 구현해야함.

3. Controller(VIew와 Model을 이어주는 다리 역할 == VIew와 Model 사이의 인터페이스 역할)
    3-1) 모델이나 뷰는 서로의 존재를 모르고 있어야하지만 컨트롤러는 모델과 뷰에 대해 알고 있어야한다.
    3-2) 모델이나 뷰의 변경내용을 받으면 각 구성 요소에게 통지해야함.
    3-3) 사용자가 어플리케이션을 조작하여 발생하는 변경 이벤트들을 처리하는 역할을 함.
    3-4) 컨트롤러가 갖춰야할 규칙
        3-4-1) 모델이나 뷰에대해 알고 있어야함.
        3-4-2) 모델이나 뷰의 변경을 모니터링 해야함.
```

## Model

```
1. 컨트롤러에서 데이터를 생성하여 뷰에 전달하는 역할.
2. HashMap형태로 Key : Value 값을 저장함.
3. model.addAttribute(key, Value) 를 통해 View로 데이터를 전달.
```

## Model 사용법

```java
//Model 객체를 파라미터로 받아서 데이터를 뷰로 넘길 수 있다.
@RequestMapping("/board/view")
public String view(Model model) {

    // 데이터만 설정이 가능
    model.addAttribute("id", "hongku");

    return "board/view";
}

//Model 객체를 파라미터로 받는다, 스프링은 그 값을 뷰쪽으로 넘겨준다.
model.addAttribute("변수이름", "변수에 넣을 데이터값");

//뷰(.jsp 파일)에서는 ${}를 이용해서 값을 가져온다.
${변수이름}

```
